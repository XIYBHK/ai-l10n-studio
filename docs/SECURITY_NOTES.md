# 安全说明

## 当前密钥存储方案

- 项目当前采用“公开配置 + 独立 secrets 文件”的拆分存储。
- 运行时主路径使用 `ConfigDraft`，会把公开配置与密钥分开写入。
- 前端读取配置摘要时，只暴露掩码后的密钥预览，不返回完整 API Key。
- `ConfigManager` 仅保留为兼容/工具型入口，存储行为也已对齐到同一套 secrets 拆分方案。

## 为什么暂时不直接上系统凭据库

- 当前项目是单机桌面工具，不是多租户服务，也不是浏览器 SaaS。
- 现有方案已经避免了最主要的问题：API Key 混入普通配置、日志或前端返回值。
- 如果现在就引入跨平台 keychain / credential manager，会明显增加：
  - 平台兼容与迁移成本
  - 便携模式支持复杂度
  - 测试与故障排查成本
  - 首次配置与迁移失败的用户支持成本

## 什么时候值得升级

以下情况出现时，建议再升级到系统凭据库：

- 面向企业用户或受审计环境分发
- 需要支持共享终端 / 多用户设备
- 需要通过更严格的安全合规检查
- 项目开始保存更多高价值凭据，而不只是 AI API Key

## 建议结论

- 对当前阶段，这不是必须项。
- 直接上系统凭据库存储，现阶段更像“可做但不急做”，属于中期增强而不是当下阻塞项。
- 当前更合适的策略是：保持拆分存储、避免泄露到公开配置与日志、继续最小化权限。
